# Docstring Style

В проекте используется единый стиль docstring'ов: `Google style`.

Цель этого документа:
- зафиксировать один понятный формат для всего репозитория;
- сделать код читаемым через год-два без "угадывания" контракта;
- снизить хаос от смешения `reST`, `NumPy`, произвольных комментариев и
  устных договорённостей.

## Почему выбран Google style

Для этого проекта важнее:
- быстро читать прикладной код;
- понимать контракты асинхронных методов;
- видеть побочные эффекты, очереди, исключения и ограничения.

`Google style` для этого подходит лучше всего:
- короче `NumPy style`;
- проще глазами, чем `reST`;
- хорошо сочетается с type hints;
- легко применять постепенно, без большой миграции.

## Общие правила

1. Один стиль на весь проект.
2. Docstring должен объяснять контракт и смысл, а не пересказывать код.
3. Типы не дублируются в тексте без необходимости, если они уже есть в
   аннотациях.
4. Если у функции есть побочные эффекты, это нужно явно писать.
5. Для асинхронных методов обязательно указывать:
   - что они ждут;
   - что публикуют;
   - что может остановить цикл;
   - какие есть важные ограничения.
6. Для приватных методов docstring нужен только если логика неочевидна.
7. Для простых property и очевидных геттеров достаточно одной короткой фразы.
8. Если метод может бросать исключение по контракту, это отражается в `Raises`.
9. Если есть важные допущения по времени, потокобезопасности, очередям или
   состоянию объекта, это указывается в `Notes`.

## Структура docstring

### Для модуля

В начале файла допустим длинный module-level docstring, если модуль:
- нетривиальный;
- содержит бизнес-логику;
- используется как точка расширения;
- требует объяснения архитектурного контекста.

Рекомендуемая структура:
- что делает модуль;
- какие сущности в нём основные;
- где его зона ответственности;
- что в модуле намеренно не решается.

### Для функции или метода

Рекомендуемый шаблон:

```python
def func(arg1: int, arg2: str | None = None) -> bool:
    """Short summary in imperative or descriptive form.

    Longer explanation if the behavior is not obvious from the signature.

    Args:
        arg1: Meaning of the first argument.
        arg2: Meaning of the second argument and important constraints.

    Returns:
        What exactly is returned and in what situations.

    Raises:
        ValueError: When input is invalid.

    Notes:
        Additional behavior, caveats, side effects, or invariants.
    """
```

### Для класса

Рекомендуемый шаблон:

```python
class Example:
    """Short description of the class responsibility.

    Describe what the class owns, what problem it solves, and what important
    invariants or lifecycle assumptions exist.
    """
```

Если класс сложный, допустимо дополнительно описывать:
- жизненный цикл;
- внутреннюю модель состояния;
- ожидаемый сценарий использования;
- ограничения потокобезопасности.

## Когда использовать секции

- `Args`: почти для всех публичных функций и методов.
- `Returns`: если результат не очевиден из названия или есть несколько
  сценариев возврата.
- `Raises`: если исключения являются частью нормального контракта.
- `Notes`: если есть важные ограничения, сайд-эффекты, асинхронные детали,
  особенности времени, блокировок, очередей, кэша, повторных подключений.
- `Examples`: только там, где пример реально ускоряет понимание.

## Чего не делать

Не надо:
- писать "Function for..." или "Method that does...";
- дублировать типы из аннотаций без новой информации;
- расписывать очевидный код;
- использовать разные стили секций в разных файлах;
- смешивать `Args` и русские самодельные списки внутри одного docstring;
- описывать исторические причины, если они не влияют на поддержку сейчас.

Плохо:

```python
def reset(self) -> None:
    """Reset method."""
```

Лучше:

```python
def reset(self) -> None:
    """Reset the accumulated window state and forget previous timestamps."""
```

## Язык

В проекте сейчас много русскоязычных комментариев. Это допустимо.

Правило простое:
- если модуль уже русскоязычный, новые docstring'и можно писать на русском;
- но стиль секций должен оставаться единым: `Args`, `Returns`, `Raises`,
  `Notes`, `Examples`;
- внутри одного docstring не смешивать хаотично русский, английский и разные
  форматы оформления.

## Порядок миграции

Не нужно массово переписывать весь репозиторий за один раз.

Нормальный порядок:
1. Зафиксировать стиль этим документом.
2. Выбрать 2-3 эталонных модуля.
3. При каждом изменении файла заодно приводить его docstring'и к стандарту.
4. Отдельно дочищать старые ключевые модули, если это оправдано.

Приоритетные файлы для миграции:
- `modules/arbitrage_manager.py`
- `modules/exchange_instrument.py`
- `modules/arbitrage_pairs.py`
- `modules/ArbitrageRootClass.py`
- `modules/utils.py`
- `modules/exception_classes.py`

## Критерий качества docstring

Хороший docstring отвечает на вопросы сопровождающего:
- зачем это существует;
- что принимает;
- что возвращает;
- какие есть ограничения;
- какие побочные эффекты;
- что может пойти не так.

Если после чтения docstring всё ещё нужно лезть в реализацию, чтобы понять
контракт, значит docstring слабый.
